모순 및 오류

8건 수정 완료

FIXED safe_revert() 파괴적 동작

문제: git checkout -- .로 전체 working tree를 삭제. scope 밖 사용자 작업까지 날림.

수정: git checkout HEAD -- $SCOPE_FILES로 scope 파일만 복원.

FIXED Phase 0 git add -A

문제: git init 시 git add -A 사용 — 자체 규칙("절대 git add -A 금지") 위반.

수정: 구체적 파일만 staging + 경고 코멘트 추가.

FIXED Metric/Verify 이중 파라미터

문제: 둘 다 필수 파라미터인데 관계가 불명확. 루프에서는 verify_cmd만 사용.

수정: Verify(필수) + metric_extraction(반필수, 기본값: 자동 추출)으로 재정의.

FIXED 상태 출력 주기 불일치

문제: SKILL.md는 "5회마다", results-logging.md는 "10회마다".

수정: 5회=1줄 brief status, 10회=summary block으로 이원화.

FIXED PIVOT 트리거 기술 모호

문제: SKILL.md가 REFINE(3회) 단계를 누락하고 "5회→pivot"만 기술.

수정: "3회→REFINE, 5회→PIVOT, 2 PIVOTs→Web Search, 3 PIVOTs→Bold" 전체 명시.

FIXED 메트릭 추출 전략 불일치

문제: "last number on last line" vs grep | tail -1(모든 라인의 마지막 매치) — 다른 결과 가능.

수정: grep -oE '[0-9]+\.?[0-9]*' | tail -1 (전체 출력의 마지막 숫자)로 통일.

FIXED baseline/current_best 용어 혼용

문제: "baseline"이 원래 시작값과 현재 최고값을 모두 의미하는 데 사용.

수정: baseline=원래 시작값(불변), current_best=현재 최고값(keep마다 갱신).

FIXED .gitignore 자동 설정 누락

문제: 결과/상태/레슨 파일을 gitignore한다고 명시하지만 실제 추가 로직 없음.

수정: Phase 0에 .gitignore 자동 체크/추가 단계 신설.

위험 요소

CRITICAL Goodhart's Law

"측정이 목표가 되면 좋은 측정이 아니게 된다." 에이전트가 의미 없는 테스트 추가, 기능 삭제로 복잡도 감소 등 메트릭만 올리는 행동 가능.

대응: Guard 명령 필수 설정. 메트릭과 Guard의 이중 검증.

CRITICAL Git 히스토리 오염

experiment + revert 커밋이 대량 축적. 팀 프로젝트 main 브랜치에서 사용하면 히스토리 재앙.

대응: 반드시 실험 전용 브랜치에서 사용. 완료 후 squash merge.

HIGH Context window 고갈

Unbounded 모드에서 20-30 iteration이면 context 한계 도달. Session resume 있으나 전환 시 추론 품질 저하.

대응: Bounded iterations(20-30)으로 시작. 결과 확인 후 연장.

HIGH 비용 폭주 (Unbounded)

verify가 느리거나 비용이 드는 경우(GPU, cloud CI) 무한 루프는 리소스 낭비.

대응: Bounded 모드 권장. verify 30초 이내 목표.

MEDIUM Scope 경계 교착

scope 밖 파일 변경이 필요한데 규칙이 금지. 5+ discard → PIVOT → 또 discard 반복.

대응: scope를 약간 넓게 설정. 또는 중간에 scope 조정.

MEDIUM Pre-commit hook 충돌

formatter/linter hook이 실험 코드를 거부 → hook-blocked 빈발 → 유효 iteration 감소.

대응: hook이 자동 수정하는 유형이면 문제 없음. 차단형은 주의.

MEDIUM 미묘한 버그 도입

메트릭은 통과하지만 edge case에서 깨지는 코드. Guard가 방어하나 100%는 아님.

대응: 실험 완료 후 코드 리뷰 필수. /autoresearch:ship으로 PR 생성.

LOW Noisy metric 오판

타이밍 기반 메트릭 등에서 랜덤 변동으로 잘못된 keep/discard 결정.

대응: min_delta 설정, confirmation run, multi-run averaging.

메트릭 향상

NEW — 구현 완료

Level 1: Direct

목표 자체가 숫자. 별도 설계 불필요.

  • 테스트 통과 수
  • 에러 카운트
  • 번들 사이즈 (bytes)
  • 실행 시간 (ms)

상관도: 1:1 (완벽)

Level 2: Proxy

주관적 목표의 기계적 대리 지표.

  • "가독성" → cognitive complexity
  • "테스트 품질" → mutation score
  • "API 안정성" → fault injection 통과율
  • "성능" → hyperfine median p99

상관도: 높음 (proxy 선택에 의존)

Level 3: Composite

복합 목표를 가중 합산. 스크립트로 생성.

  • metric-test-quality.sh
    = 0.6×pass_rate + 0.4×coverage
  • metric-code-health.sh
    = 100 - 3×errors - 1×lint + 0.3×tests
  • metric-perf-score.sh
    = latency_score × weight

상관도: 조정 가능 (가중치 튜닝)

Anti-Gaming 3계층 방어

계층메커니즘방어 대상상태
Layer 1 Guard 명령 "메트릭은 올랐는데 다른 게 깨졌다" (regression) 기존
Layer 2 Metric Sanity Check 비정상 점프 감지 (3× 평균 delta 초과 시 재검증) NEW
